*Дорожня карта переходу від доведеного MVP до повної гри.*
*Створено 2026-06-20. Спирається на [GDD](../GDD_drone_factory.md) і [progress.md](../progress.md).*

> **Розворот 2026-06-20:** проєкт переходить з рантайм-3D (Babylon) на **2D (Excalibur)**.
> Причини (з обговорення): загальна **складність 3D** і **виробництво моделей** (Blender-pipeline) — надто дорого для соло-розробки.
> Ядро гри (`gameState`, `kits`, `upgrades`, `config`, save, тести, міні-гра паяння) **не залежить** від рушія і переноситься без змін — міняється лише шар `src/scene/`.
> Це не «викидання роботи»: реальних `.glb`-моделей ще не зроблено, у 3D стоять процедурні примітиви. Розворот робиться рівно в точці, де 3D мав почати дорожчати.

> **Уточнення напряму 2026-06-20 (друге):** псевдо-ізо 3⁄4 → **top-down (вид зверху, у стилі Prison Architect)** з **повноцінним персонажем-робітником**, що ходить кімнатою і фізично виконує дії (бере коробку, сідає за стіл, паяє).
> Постійна панель UI прибирається — лишається **мінімальний HUD** (гроші + кнопка-магазин); решта взаємодії **diegetic, через тапи по об'єктах** (тап по коробці → принести; тап по столу → паяти).
> **Керування робітником:** спершу тап = команда, робітник — виконавець (гравець лишається «руками» в міні-грі паяння); пізніше купується апгрейд, що перемикає робітника на **автономний** режим. Це **та сама `mode`-система** (`manual`/`semi`/`auto`), що вже є в `upgrades.js` — автономність стає ще одним upgrade-треком, а не новою механікою.
> Так само як розворот вище — робиться, поки в сцені стоять прямокутники-примітиви, тобто майже безкоштовно. Ядро гри знову **не зачіпається** (воно і engine-, і view-agnostic) — міняється `src/scene/` + новий шар «робітник» + перебудова UI.

## 0. Принцип: розділити чотири шари

Зараз поняття «конфіг» змішує речі з різним життєвим циклом. Розділяємо — це ключ до масштабування:

| Шар | Що це | Хто змінює | Де живе |
|---|---|---|---|
| **Content (дані гри)** | типи дронів, апгрейди, локації, спрайти | розробник/дизайнер | data-модулі (`kits.js`, `upgrades.js`, `manifest.js`) |
| **Balance (config)** | глобальні числа-knobs (стартові гроші, формула ціни, пороги поломки, тайминг міні-гри) | розробник при балансі | `src/state/config.js` |
| **Settings (гравця)** | звук, гучність, мова, гаптика, якість графіки | гравець у меню | файл `settings.json` (Capacitor) |
| **Save (прогрес)** | гроші, апгрейди, фаза, лог продажів | автоматично | файл `save.json` (Capacitor) |

**Конвенція чисел:** глобальні balance-knobs → `config.js`. Числа конкретного контент-запису (цей кіт коштує 72; цей рівень апгрейду коштує 150) → у data-реєстрі цього контенту. Жодних магічних чисел у логіці.

> Цей шар розворот **не зачіпає** — він ортогональний до рендера.

## 1. Asset pipeline — 2D-спрайти

> Детальні ТЗ на кожен спрайт — у [asset_specs.md](../asset_specs.md) (потребує 2D-проходу: габарити в пікселях, кадри, якорі-зміщення замість полігонажу/named-вузлів).

**Рушій: [Excalibur.js](https://excaliburjs.com/)** — повноцінний 2D-двіжок (scene graph, Actors, input, actions/анімації, resource loader). Лягає на поточну архітектуру майже 1:1:

| Babylon (було) | Excalibur (стане) |
|---|---|
| `Mesh` / `TransformNode` | `Actor` |
| `mesh.position` (Vector3) | `actor.pos` (Vector2) |
| `setEnabled(bool)` | `actor.graphics.visible` / add↔remove зі сцени |
| `ActionManager` OnPick | `actor.on('pointerup', …)` |
| `Animation` (коробка на стіл) | `actor.actions.moveTo(…)` / easing |
| `SceneLoader.ImportMeshAsync` + `.glb` | `ex.ImageSource` + PNG/спрайтшит |
| `getModel(key)` → mesh \| null | `getSprite(key)` → `Sprite` \| null |

**Чому Excalibur дає два наші інваріанти «безкоштовно»:**
- **Процедурний fallback** — `Actor` малюється як кольоровий прямокутник (`color`) без спрайту. Поточний «програмерський» вигляд (box-дрон, стіл, коробка) переноситься 1:1; спрайти підкладаються пізніше по одному. Принцип «нуль асетів = працює як зараз» зберігається.
- **Async-UX завантаження** — у Excalibur є вбудований `Loader` з прогрес-баром. Те, що для `.glb` робили руками (під-етап 2.4), дає рушій із коробки.

**Чесний нюанс:** Excalibur 2.x рендерить через WebGL (canvas2d-фолбек прибрано). Тобто «WebGL у WebView» формально лишається — але для 2D-навантаження це копійки проти 3D. WebGL-перформанс у драйвери розвороту не входив; якщо колись стане проблемою — fallback-варіант PixiJS/чистий canvas (втрата зручностей рушія, більше ручного коду).

**Обраний підхід до арту (гібрид із єдиним стилем):**
- Брати готові **CC0 2D-ассети** де можливо — **Kenney.nl** (величезні паки: предмети, топ-даун/ізо, UI), єдиний стиль, безкоштовно, **без Blender**.
- Те, чого немає або що має бути впізнаваним (дрони, коробка доставки, інструменти) — малювати власні спрайти за окремим spec sheet.
- **Єдиний стиль** — обов'язкова умова: спільна палітра, однаковий піксельний/векторний підхід, узгоджений ракурс для всіх ассетів (своїх і запозичених).

### Формат і технічні правила
- **Формат: `.png`** (з прозорістю), за потреби **спрайтшит** (атлас кадрів) для багатокадрових елементів/анімацій.
- **Стиль: low-detail 2D** під єдиний ракурс (див. §6 — відкрите рішення про вид: збоку / 3⁄4 / псевдо-ізо).
- **Конвенції (до них прив'язується код):**
  - Точка постановки = `anchor` спрайту (origin), зазвичай центр-низ для предметів на столі.
  - Розміри в пікселях; один логічний піксель-юніт узгоджений зі світовими координатами сцени.
  - Осмислені ключі (`mini_drone`, `delivery_box`, `workbench`) — код адресує спрайт за ключем.
  - Якорі як 2D-зміщення (`anchor_box`, `anchor_drone`, `anchor_solder_1..N`) — точки постановки/пайки відносно origin спрайту.
  - Спільна палітра кольорів для єдиного стилю (і для fallback-прямокутників).

### Де зберігати ассети
```
src/assets/
  sprites/        # .png — кожен спрайт окремий файл (mini_drone.png, delivery_box.png, ...)
  sprites/sheets/ # спрайтшити-атласи (багатокадрові / анімовані)
  CREDITS.md      # ліцензії та автори запозичених CC0-ассетів (Kenney тощо)
```
- Vite кладе спрайти у білд як статику; лоадер вантажить за URL (через `public/sprites/` або `import.meta.url`).
- Vendored CC0-спрайти — у тому ж `sprites/`, їхні джерела й ліцензії — у `CREDITS.md`.
- Тека `src/assets/models/` і `public/models/` з 3D-ерою — **видаляються** при міграції (крок M5).

### Список спрайтів під першу ітерацію (заміна процедурних прямокутників)
`mini_drone.png`, `delivery_box.png`, `workbench.png`, `soldering_iron.png` (+ варіанти рівнів), оточення кімнати (CC0 Kenney або процедурні прямокутники). На кожен власний спрайт — окремий spec sheet (ТЗ: габарити в px, якорі, кадри, кольори).

### Код: реєстр + лоадер
- `src/assets/manifest.js` — реєстр `SPRITES = { [key]: { url, anchors } }`, single source of truth (якорі — 2D-зміщення).
- `src/scene/loader.js` — обгортка над Excalibur `ImageSource`/`Loader`: вантажить усе з маніфесту в кеш `Map<key, Sprite>`, показує прогрес; відсутній/битий файл — `try/catch`, лог-ворнінг, кеш ставить `null` (fallback на прямокутник), решта вантажиться далі.
- `src/scene/spriteCache.js` — чиста логіка кешу (`getSprite`, `getAnchor`), **нуль імпортів Excalibur**, тестується в Node (аналог поточного `modelCache.js`).
- `scene.js`: `buildDrone()`/`buildBox()` спершу пробують `getSprite(key)`; якщо `null` — малюють кольоровий `Actor`-прямокутник. **Критично — лишити процедурний fallback.**

## 3. Settings і Save у файл (Capacitor) — розворот не зачіпає

Розділити те, що зараз злите в `storage.js` (localStorage):
- **Save (прогрес):** `@capacitor/filesystem`, `save.json` у `Directory.Data`. Версіонування вже є (`SAVE_VERSION`). Веб-фолбек на `localStorage`. Інтерфейс `storage.js` не міняється — лише реалізація.
- **Settings (гравця):** окремий модуль `src/save/settings.js` → `settings.json` (звук, гучність, мова, гаптика, якість графіки). Не скидається при `clearSave`.
- **Config (баланс):** лишається в коді (`config.js`).
- Адаптер `platformStorage` з `read/write(name)`: під капотом Filesystem (нативно) або localStorage (веб).

## 5a. Стадія 2-bis — Top-down 2D кімната + робітник (Excalibur)

**Мета стадії:** замінити шар `src/scene/` (Babylon) на Excalibur **і** перейти на top-down (PA-style) сцену з персонажем-робітником та diegetic-взаємодією. Ключовий інваріант — **ядро (FSM, кіти, апгрейди, save, міні-гра), тести й Android-білд не ламаються**; гра лишається повноцінно грабельною на кожному під-етапі.

**Публічний контракт сцени (міняється мінімально — `main.js` чіпаємо точково):**
- `initScene(canvas, { onBoxPicked, onSolderRequested, onLoadProgress }) → Promise<refs>` — додається `onSolderRequested` (тап по столу → відкрити міні-гру).
- `updateScene(refs, phase)` — синхронізує сцену й робітника з фазою FSM.
- `refs.engine.getFps()` (для debug-лічильника FPS) — Excalibur-аналог.

**Шари (новий поділ):**
- `scene/scene.js` — top-down діорама (підлога, стіл, двері, лампа) + об'єкти (коробка, дрон).
- `scene/worker.js` — **новий**: Actor-робітник + його власний FSM (`idle`→`walkTo`→`pick`→`carry`→`atBench`→`solder`). Читає `mode` (`manual`/`semi`/`auto`) — на `auto` діє без тапів. Pure-логіка FSM (без Excalibur) тестується в Node.
- `ui/` — мінімальний HUD (гроші + кнопка-магазин), модалка міні-гри паяння, заглушка-модалка магазину.

### M1 — Excalibur in, Babylon out

**Обсяг:** `npm i excalibur`; прибрати `@babylonjs/*` із залежностей. Переписати каркас `initScene`: `new ex.Engine({ canvasElement: canvas, … })`, порожня сцена з фоновим кольором (поточний `#0f0f1c`-ish), `engine.start()`. Зберегти сигнатуру (повертає Promise з `refs`, де є `engine`). `updateScene` поки no-op або базова заглушка. `main.js` не чіпати (контракт той самий); `getFps` → `engine.clock`/`ex.Engine` FPS API.

**DoD:** `npm run dev` показує порожню сцену на canvas без Babylon у бандлі; `npm run build` і `npm test` зелені; debug-FPS-лічильник показує число.

**Гейт:** чи рушій стартує на існуючому `<canvas id="game-canvas">` без конфлікту з DOM-оверлеєм UI? Чи контракт `initScene/updateScene` лишився незмінним?

### M2 — Лоадер + spriteCache з graceful fallback

**Обсяг:** `spriteCache.js` (чистий, без Excalibur-імпортів): `getSprite(key) → Sprite|null`, `getAnchor(key, name) → {x,y}|null`, три стани кешу (`undefined`/`null`/`Sprite`) — як у поточному `modelCache.js`. `loader.js`: `loadSprites(onProgress)` через `ex.ImageSource` + `ex.Loader`; відсутній/битий файл → `try/catch`, ворнінг, `setFailed(key)`, не валить решту. `manifest.js` → `SPRITES = { [key]: { url, anchors } }` (якорі — 2D-зміщення).

**DoD:** `loadSprites` резолвиться навіть коли частина/всі файли відсутні; `getSprite('mini_drone')` повертає `Sprite` якщо файл є, інакше `null`; нуль unhandled-rejection; юніт на fallback-гілку `getSprite` (Node, без WebGL) зелений.

**Гейт:** чи відсутній файл = `null`, а не краш? Чи `spriteCache` тестується в Node без WebGL-контексту?

### T1 — Top-down перекомпонування кімнати (без персонажа)

**Обсяг:** рефактор `scene.js` на Excalibur Actors у **вид зверху**: підлога, стіл, двері (точка появи коробки), лампа. Кожен — `Actor` з `color` (кольоровий прямокутник, поточна палітра) поки нема спрайту. Коробка й дрон — Actors на своїх місцях. `updateScene(refs, phase)`: видимість коробки (DELIVERY) і дрона (ASSEMBLY/READY) через `graphics.visible`, ресет коробки до дверей поза DELIVERY. Поки **без персонажа** — коробка на тап летить на стіл `actions.moveTo` (як зараз), щоб гра лишалась грабельною до T2.

**DoD:** top-down кімната з нулем спрайтів грабельна й поведінково ідентична поточній (коробка→стіл→збірка→міні-гра→продаж). Жодних регресій у ядрі/тестах.

**Гейт:** чи top-down-компонування читається (стіл/двері/коробка пізнавані прямокутниками)? Чи фази FSM коректно керують видимістю?

### T2 — Робітник-аватар + його FSM (тап = команда, manual)

**Обсяг:** `scene/worker.js` — Actor-робітник + **pure FSM** (`idle`→`walkTo`→`pick`→`carry`→`atBench`→`solder`→`idle`), тестований у Node без Excalibur. Рух — `actions.moveTo` по waypoint-ах (підлога відкрита, **без A*/pathfinding** на цьому етапі). Тап по коробці = команда «принеси» → робітник іде до дверей, бере, несе на стіл → тригерить `onBoxPicked`. Тап по столу (коли є що паяти) = команда «паяй» → робітник сідає → `onSolderRequested` (T3 відкриє міні-гру). worker-FSM **підпорядкований** game-FSM: команди дозволені лише у відповідній фазі; завершення фізичної дії синхронізується з переходами `gameState`.

**DoD:** робітник видимо ходить і виконує цикл доставки+підготовки до пайки за тапами; неможливо віддати команду в невідповідній фазі; ядро/save/тести не зламані; worker-FSM має юніт-тести.

**Гейт:** чи робітник надійно доходить і не «застрягає» між станами? Чи worker-FSM лишається підпорядкованим game-FSM (немає рассинхрону)?

### T3 — Interaction-driven UI (мінімальний HUD)

**Обсяг:** прибрати постійну панель `domUI`. Лишити **мінімальний HUD**: лічильник грошей + кнопка «Магазин». Кнопка відкриває **заглушку-модалку магазину** (поки список кітів + кнопка «Замовити», візуал магазину — окремий крок пізніше). Міні-гра паяння → **модалка**, що відкривається по `onSolderRequested` (тап по столу) і ховається після завершення. Статус/підказки фаз — мінімальні, diegetic де можливо.

**DoD:** на екрані постійно лише гроші + кнопка-магазин; замовлення йде через модалку магазину; пайка відкривається тапом по столу; повний цикл грабельний без старої панелі; апгрейди доступні (поки в модалці магазину або окремій модалці — не в постійному UI).

**Гейт:** чи весь цикл проходимо лише через тапи по світу + HUD? Чи гравцю зрозуміло, де гроші й що робити далі без постійної панелі?

### T4 — Авто-режим робітника через upgrade-трек

**Обсяг:** новий (або розширений) трек в `upgrades.js` з рівнями `mode: manual → semi → auto` для робітника (дзеркалить наявну mode-механіку паяльника). `worker.js` читає поточний `mode`: на `manual` чекає тапів; на `semi` — частина дій авто; на `auto` — робітник сам виконує доставку й підготовку без тапів (пайка лишається за гравцем доти, доки auto-паяльник теж не куплено). Жодної нової FSM — лише гілкування входів worker-FSM за `mode`.

**DoD:** після купівлі авто-апгрейду робітник виконує цикл без тапів; до купівлі — поведінка T2 (ручні команди); перемикання режиму не ламає FSM і зберігається в save.

**Гейт:** чи `mode` коректно гілкує worker-FSM без дублювання логіки? Чи автономний цикл стабільний на довгій дистанції?

### M4 — Перший спрайт + swap

**Обсяг:** покласти один реальний спрайт (тимчасово — будь-який PNG, напр. `delivery_box.png`), додати запис у `manifest.js`. Перевірити, що `getSprite` віддає його і `Actor` показує спрайт замість прямокутника в тій самій позиції з тим самим origin/якорем.

**DoD:** підкладений спрайт заміняє прямокутник без інших змін коду; origin/якір сідає там, де стояв прямокутник (не зсунутий/іншого масштабу); прибраний файл → знову прямокутник.

**Гейт:** чи swap прямокутник↔спрайт справді безшовний (позиція, масштаб, z-порядок)?

Перейменувати `modelKey`→`spriteKey` у `kits.js`. Оновити тести під нові імена. Перевірити `npm run build`, `npm test`, `npm run android` (білд + sync) і дебаг-FPS.

**DoD:** нуль згадок Babylon/`.glb`/`.blend` у коді й залежностях; усі тести зелені; Android-білд збирається й запускається; бандл помітно менший (3D-рушій прибрано).
