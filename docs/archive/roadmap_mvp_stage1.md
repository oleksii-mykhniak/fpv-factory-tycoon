# Roadmap — Стадія 1 (MVP «Квартира»): розбивка на під-етапи

> ⚠️ **АРХІВ (закрито 2026-06-21).** Усі під-етапи 1.0–1.8 виконані — див. [progress.md](../progress.md).
> Зберігається як історія. Актуальна дорожня карта → [plan_demo_ready.md](../plan_demo_ready.md).

*Деталізує Стадію 1 з [GDD](../GDD_drone_factory.md) у конкретні, тестовані шматки коду.*

## Принципи розбивки

1. **Стан гри окремо від рендеру** (архітектурний принцип GDD). Уся логіка — у чистому JS-модулі, що нічого не знає про DOM чи Babylon.
2. **3D — в останню чергу.** Весь ігровий цикл доводимо на простому DOM-UI. Дешево ітерувати до Babylon, дорого — після.
3. **Кожен під-етап = граний/тестований шматок** зі своїм definition of done. Не починаємо наступний, поки не закрили попередній.

## Цільова структура проєкту

```
src/
  state/      # чистий стан + логіка (без DOM/Babylon), з юніт-тестами
  ui/         # DOM-оверлей: магазин, HUD, міні-гра паяння
  scene/      # Babylon-сцена (додається на 1.7)
  save/       # серіалізація стану
```

## Під-етапи

### 1.0 — Каркас і тулчейн
- Vite-проєкт, структура папок `state/ ui/ scene/ save/`.
- Порожня сторінка з канвасом-плейсхолдером.
- **DoD:** `npm run dev` відкриває сторінку, hot-reload працює.

### 1.1 — Чистий стан (ядро) ⭐ найважливіший
- JS-модуль single source of truth: гроші, інвентар, прогрес циклу.
- FSM циклу: `ЗАМОВЛЕННЯ → ДОСТАВКА → ЗБІРКА → ПРОДАЖ`.
- Функції: `orderKit`, `receiveDelivery`, `startAssembly`, `recordSolderPoint`, `finishAssembly`, `sell`.
- Формула ціни з GDD: `ціна = база × (0.6 + 0.7 × якість) × множник_прокачки`.
- Нуль залежностей від DOM/Babylon.
- **DoD:** юніт-тести проганяють повний цикл у коді, гроші змінюються вірно; невалідні переходи FSM відхиляються.
- *Цей модуль житиме і в прототипі, і у фінальній грі.*

### 1.2 — DOM-луп (текстовий прототип = гейт Стадії 0)
- Кнопки, прив'язані до функцій стану: магазин, баланс $, «доставити», «зібрати» (якість поки заглушка/рандом), «продати».
- **DoD:** можна зіграти 10+ циклів кліками, баланс росте.
- **Гейт:** чи хочеться повторити цикл без зовнішнього стимулу?

### 1.3 — Міні-гра паяння (серце раннього геймплею)
- Реакційна гра: повзунок по треку, зелена зона (центр = ідеал), тап.
- Якість кожної точки → загальна оцінка збірки. Швидкість повзунка трохи зростає з кожною точкою.
- Замінює заглушку якості з 1.2.
- **DoD / гейт:** промах відчувається прикро, ідеальна пайка — приємно («вага» міні-гри).

### 1.4 — Поломка
- Холодна пайка (переробити точку, втрата часу) / перегрів (деталь спалено → гілка реордеру).
- Підключено до промахів у міні-грі; співвідношення тюниться.
- **DoD:** форсовані промахи коректно ведуть в обидві гілки; поломка перериває цикл одного дрона, не весь прогрес.

### 1.5 — Дерево апгрейдів (вісь A: автоматизація праці)
- Ручний тап → кращий паяльник (ширша зелена зона, менший шанс поломки) → напівавтомат (1 тап/збірка) → автопаяльник (паяє у фоні, повільніше й трохи гірше).
- Купівля за гроші.
- **DoD:** кожен апгрейд відчутно міняє поведінку.
- **Гейт (дизайн Стадії 1):** автопаяльник «зручніший, але не строго кращий» за майстерну ручну пайку — мотивація паяти руками зберігається.

### 1.6 — Збереження
- Серіалізація стану: localStorage на вебі → `@capacitor/preferences` під нативку.
- **DoD:** перезавантаження сторінки зберігає прогрес.

### 1.7 — Babylon 3D + доставка коробки
- Low-poly сцена: кімната, стіл, коробка, дрон.
- Взаємодія доставки: тап коробку → персонаж/камера несе → тап стіл → кладе.
- Міні-гра паяння інтегрується як HUD-оверлей над сценою.
- **DoD:** прийнятний FPS у браузері; взаємодія доставки працює; луп проходиться у 3D.

### 1.8 — Capacitor + тест на залізі
- Пакування під Android/iOS через Capacitor.
- Запуск на реальному (бажано слабкому) телефоні.
- **DoD / технічний гейт Стадії 1:** гра запускається, тримає прийнятний FPS у WebView, прогрес зберігається між запусками.

## Як гейти Стадії 1 лягають на під-етапи

- **Дизайн-гейт** (автопаяльник зручніший, але не строго кращий) → перевіряємо на **1.5**.
- **Технічний гейт** (Babylon тримає FPS у WebView) → перевіряємо на **1.8**.
- Якщо слабкий дизайн-сигнал — перебалансовуємо на 1.3–1.5, *до* 3D.
- Якщо слабкий технічний сигнал — спрощуємо сцену на 1.7–1.8, *до* нарощення контенту.
