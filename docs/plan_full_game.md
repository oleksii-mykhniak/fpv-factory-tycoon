## 0. Принцип: розділити чотири шари

Зараз поняття «конфіг» змішує речі з різним життєвим циклом. Розділяємо — це ключ до масштабування:

| Шар | Що це | Хто змінює | Де живе |
|---|---|---|---|
| **Content (дані гри)** | типи дронів, апгрейди, локації, моделі | розробник/дизайнер | data-модулі (`kits.js`, `upgrades.js`, `manifest.js`) |
| **Balance (config)** | глобальні числа-knobs (стартові гроші, формула ціни, пороги поломки, тайминг міні-гри) | розробник при балансі | `src/state/config.js` |
| **Settings (гравця)** | звук, гучність, мова, гаптика, якість графіки | гравець у меню | файл `settings.json` (Capacitor) |
| **Save (прогрес)** | гроші, апгрейди, фаза, лог продажів | автоматично | файл `save.json` (Capacitor) |

**Конвенція чисел:** глобальні balance-knobs → `config.js`. Числа конкретного контент-запису (цей кіт коштує 72; цей рівень апгрейду коштує 150) → у data-реєстрі цього контенту. Жодних магічних чисел у логіці.

## 1. Asset pipeline — 3D-моделі

> Детальні ТЗ на кожну модель і текстури — у [asset_specs.md](asset_specs.md).

**Обраний підхід (гібрид із єдиним стилем):**
- Брати готові **CC0 low-poly** ассети де можливо (Kenney, Quaternius) — меблі, оточення, декор.
- Те, чого немає або що має бути впізнаваним (дрони, коробка доставки, інструменти) — **моделювати в Blender** за окремим spec sheet на кожну модель.
- **Єдиний стиль** — обов'язкова умова: один підхід до полігонажу, палітри й матеріалів для всіх ассетів (своїх і запозичених). Запозичені моделі за потреби перефарбовуємо під спільну палітру в Blender.

### Формат і технічні правила
- **Формат: `.glb`** (binary glTF) — рідний для Babylon, один файл = меш+матеріали+анімації.
- **Стиль: low-poly, baked lighting** (ризик WebGL у WebView — див. GDD §9).
- **Конвенції Blender** (до них прив'язується код):
  - Origin у точці постановки (дрон/коробка — центр низу; меблі — центр підлоги).
  - Метри, `Apply Transform` перед експортом (scale = 1).
  - Осмислені імена під-мешів (`motor_FL`, `esc_1`) — код адресує їх за іменем.
  - Empty-якорі: `anchor_solder_1..N` (точки пайки/іскри), `anchor_box` (місце коробки на столі).
  - Спільна палітра кольорів-матеріалів для єдиного стилю.

### Де зберігати ассети
```
src/assets/
  models/        # .glb — кожна модель окремий файл (mini_drone.glb, delivery_box.glb, ...)
  textures/      # спільні атласи / baked-текстури
  models/_blender/  # вихідні .blend (не входять у білд, тримаємо в репо для правок)
  CREDITS.md     # ліцензії та автори запозичених CC0-ассетів
```
- Vite кладе `src/assets/models/*.glb` у білд як статику; лоадер вантажить за URL.
- Vendored CC0-моделі — у тому ж `models/`, їхні джерела й ліцензії — у `CREDITS.md`.

### Список моделей під Стадію 1 (заміна процедурних примітивів)
`mini_drone.glb`, `delivery_box.glb`, `workbench.glb`, `soldering_iron.glb` (+ варіанти рівнів), оточення кімнати (CC0 або процедурне). На кожну власну модель — окремий spec sheet (ТЗ: габарити, named-вузли, якорі, бюджет полігонів, кольори).

### Код: реєстр + лоадер
- `src/assets/manifest.js` — реєстр `{ key, url, anchors }`, single source of truth.
- `src/scene/loader.js` — обгортка над Babylon `AssetsManager`/`ImportMeshAsync`: вантажить усе з маніфесту в кеш `Map<key, mesh>`, показує прогрес.
- Рефактор `scene.js`: `buildDrone()` → `instantiate('mini_drone')`. **Критично — лишити процедурний fallback:** якщо `.glb` ще немає, малювати поточний box-примітив. Тоді примітиви міняються на моделі по одній, гра не ламається проміжно.

## 2. Узагальнена система контенту (data-driven) — ✅ перший крок зроблено

Замість хардкоду — реєстри даних. Новий апгрейд/кіт = новий запис, нуль змін у логіці.

**Зроблено в цьому рефакторі:**
- `src/state/upgrades.js` — реєстр `UPGRADE_TRACKS`: трек = `{ id, name, stateKey, costs[], levels[] }`. Кожен рівень несе `mode` (`manual`/`semi`/`auto`) + параметри. Хелпери `trackMaxLevel`, `levelData`, `nextCost`.
- `buyUpgrade(state, trackId)` узагальнено — працює з будь-яким зареєстрованим треком через `stateKey`.
- `src/state/kits.js` — реєстр `KIT_TYPES` винесено окремо, додано `modelKey` (ключ у маніфест ассетів).
- `main.js` гілкує збірку за `mode`, а не за номером рівня; `domUI` читає назви/ефекти/вартості з реєстру (прибрано дубль-масив).
- Тести: 22 → 46.

**Як додати новий апгрейд** (напр. «Доставка»): додати запис у `UPGRADE_TRACKS` зі своїм `stateKey`, `costs`, `levels`. UI і логіка підхоплять автоматично. Так само готуються Вісь A/B/C з GDD.

**Як додати новий тип дрону:** додати запис у `KIT_TYPES` (`cost`, `basePrice`, `solderPointCount`, `assemblySteps`, `modelKey`). Далі — власне дерево прокачки по типу (Стадія 2).

## 3. Settings і Save у файл (Capacitor)

Розділити те, що зараз злите в `storage.js` (localStorage):
- **Save (прогрес):** `@capacitor/filesystem`, `save.json` у `Directory.Data`. Версіонування вже є (`SAVE_VERSION`). Веб-фолбек на `localStorage` за детектом платформи. Інтерфейс `storage.js` не міняється — лише реалізація.
- **Settings (гравця):** окремий модуль `src/save/settings.js` → `settings.json` (звук, гучність, мова, гаптика, якість графіки). Інший життєвий цикл: не скидається при `clearSave`.
- **Config (баланс):** лишається в коді (`config.js`) — це частина білда, не файл для гравця.
- Адаптер `platformStorage` з `read/write(name)`: під капотом Filesystem (нативно) або localStorage (веб).

## 5a. Крок 2 — деталізація (Asset-реєстр + лоадер з fallback)

**Мета кроку:** сцена запитує модель за ключем; є `.glb` → вантажиться й ставиться по якорях, немає → малюється поточний процедурний примітив. Завантаження асинхронне, без «миготіння» порожньої сцени. **Ключовий інваріант усього кроку: гра з нулем `.glb` працює рівно як зараз.**

| Під-етап | Назва | Файли | Статус |
|---|---|---|---|
| 2.1 | Маніфест + структура `src/assets/` | `assets/manifest.js`, теки | 🔲 |
| 2.2 | Лоадер з кешем і graceful fallback | `scene/loader.js` | 🔲 |
| 2.3 | Інтеграція в сцену (swap примітив ↔ модель) | `scene/scene.js` | 🔲 |
| 2.4 | Async-UX завантаження (прогрес / без миготіння) | `scene/scene.js`, `main.js` | 🔲 |

> Курка-яйце: «шлях із моделлю» (2.2–2.3) валідуємо до Стадії 3 **тимчасовим тест-кубом** `mini_drone.glb` (будь-який експорт із Blender), потім замінюємо справжньою моделлю. Тест-куб у білд не комітимо.

### Під-етап 2.1 — Маніфест + структура асетів

**Обсяг:** створити теки `src/assets/{models,textures,models/_blender}` + `CREDITS.md`. Модуль `assets/manifest.js` — реєстр `MODELS = { [key]: { url, anchors: [...] } }` для `mini_drone`, `delivery_box`, `workbench`, `soldering_iron`. `url` через Vite-імпорт (`new URL('./models/x.glb', import.meta.url)`) або `?url`. Поки **жодного `.glb` може не існувати** — маніфест лише описує, що очікується.

**DoD:** `manifest.js` експортує типізований реєстр; `import` маніфесту не падає за відсутніх файлів; `npm run build` і тести зелені; `modelKey` з `kits.js` збігається з ключами маніфесту.

**Гейт:** чи однозначно реєстр відповідає на питання «який ключ → який файл → які якорі»? Чи легко додати новий запис (одна модель = один об'єкт)?

### Під-етап 2.2 — Лоадер з кешем і fallback

**Обсяг:** `scene/loader.js` — обгортка над `ImportMeshAsync`/`AssetsManager`. API:
- `loadModels(scene, manifest)` → Promise, вантажить усе, складає в `Map<key, rootMesh>`; відсутній/битий файл — `try/catch`, лог-ворнінг, **не валить** завантаження решти.
- `getModel(key)` → клон-інстанс кореня моделі або `null` якщо не завантажено.
- `getAnchor(modelInstance, name)` → `TransformNode`/позиція якоря з моделі.

**DoD:** `loadModels` резолвиться навіть коли частина (або всі) файлів відсутні; `getModel('mini_drone')` повертає меш якщо файл є, інакше `null`; жодного unhandled-rejection; модуль покритий мінімальним тестом (мок-сцена або юніт на fallback-гілку `getModel`).

**Гейт:** чи справді відсутній файл = `null`, а не краш? Чи кеш віддає інстанси, придатні для повторного показу між циклами?

### Під-етап 2.3 — Інтеграція в сцену (swap примітив ↔ модель)

**Обсяг:** рефактор `scene.js`: `buildDrone()`/`buildBox()` (і за бажання стіл) спершу пробують `getModel(key)`; якщо `null` — будують поточний box-примітив. Та сама світова позиція й `setEnabled`-логіка (`updateScene`). Якорі моделі (якщо є) виставляються замість хардкод-констант (`BOX_TABLE_POS` тощо) — з fallback на константи.

**DoD:** з нулем `.glb` гра візуально й поведінково ідентична поточній (дрон/коробка на місці, кліки працюють, анімація коробки летить туди ж); коли підкласти тест-`mini_drone.glb` — примітив заміняється моделлю в тій самій позиції без інших змін коду.

**Гейт (з GDD-стилю):** чи зберігся інваріант «нуль `.glb` = як зараз»? Чи модель сідає точно там, де примітив (а не зсунута/перевернута/іншого масштабу)?

### Під-етап 2.4 — Async-UX завантаження

**Обсяг:** `loadModels` тепер async до `runRenderLoop`/першого `draw()`. Поки вантажиться — простий прогрес-індикатор (DOM-оверлей або Babylon-текст), без показу напівпорожньої сцени. Якщо асетів нема — індикатор миттєвий/прихований. Час/таймаут завантаження — параметр у `config.js` (не магічне число).

**DoD:** на штучно сповільненому завантаженні видно прогрес і нема «миготіння» порожньої сцени; за відсутніх асетів старт миттєвий; FSM/збереження не зачеплені (стан ініціалізується незалежно від асетів).

**Гейт:** чи перехід «завантаження → гра» плавний на повільному з'єднанні? Чи не блокує лоадер ігрову логіку, якщо асет так і не прийшов?
